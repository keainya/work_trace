# 工作管理程序

一个轻量级的工作任务管理工具，帮助你追踪和管理日常工作事项。

## 核心概念

本系统围绕以下三个核心概念构建：

### 1. 工作项 (WorkItem)

工作项是任务管理的最小独立单元，代表一项具体的工作任务。

**属性：**

| 字段 | 类型 | 必须 | 说明 |
|------|------|------|------|
| 标题 | string | 是 | 工作任务的名称，如"前往X局提交材料" |
| 所属用户 | string | 是 | 从 OAuth token 解析的 `sub`，API 层自动填充 |
| 状态 | boolean | 否 | 已完成 / 未完成，默认未完成 |
| 开始时间 | datetime | 否 | 任务的计划开始时间 |
| 结束时间 | datetime | 否 | 任务的计划结束时间 |
| 提醒 | reminder | 否 | 可设置提醒，在指定时间点或提前通知 |
| 详细说明 | rich text | 否 | 自由编辑的任务详情，支持富文本 |

**功能：**
- 创建、编辑、删除工作项
- 标记工作项为"已完成"或取消完成（已完成的工作项使用删除线样式）
- 按标题搜索/筛选工作项
- 按状态（已完成/未完成）过滤
- 按起止时间排序
- 设置定时提醒

---

### 2. 子任务 (SubTask)

子任务绑定在工作项之下，用于将一个大的工作项拆分为更细粒度的待办条目。一个工作项可以添加任意数量的子任务。

**属性：**

| 字段 | 类型 | 必须 | 说明 |
|------|------|------|------|
| 所属工作项 | foreign key | 是 | 绑定的工作项 ID |
| 内容 | text | 是 | 子任务的文字描述内容 |
| 状态 | boolean | 否 | 已完成 / 未完成，默认未完成 |
| 开始时间 | datetime | 否 | 子任务的计划开始时间 |
| 结束时间 | datetime | 否 | 子任务的计划结束时间 |
| 提醒 | reminder | 否 | 可为子任务设置独立的提醒 |

**功能：**
- 添加、编辑、删除子任务
- 勾选标记完成/未完成
- 按状态（已完成/未完成）过滤
- 查看某个工作项下的所有子任务
- 子任务的完成状态不影响工作项本身

**设计说明：**
子任务不需要标题，直接用文字内容描述即可。相比工作项，子任务更轻量，侧重于快速记录和勾选。

---

### 3. 工作记录 (WorkRecord)

工作记录是对工作项所有编辑操作的自动日志，记录了每一次变更的内容和时间。

**属性：**

| 字段 | 类型 | 必须 | 说明 |
|------|------|------|------|
| 所属工作项 | foreign key | 是 | 绑定工作项 ID |
| 操作类型 | enum | 是 | 创建 / 更新 / 删除 等 |
| 变更内容 | text | 是 | 记录具体变更了哪些字段及前后值 |
| 操作时间 | datetime | 是 | 自动记录操作发生的时间戳 |

**功能：**
- 每次对工作项的编辑自动生成一条记录
- 记录变更字段及新旧值对比
- 在工作项详情界面最底部以**收起的列表**形式展示
  - 默认收起，不展开
  - 用户点击展开时才加载记录数据（懒加载）
- 按时间倒序排列（最新的在前）

**设计说明：**
工作记录是系统自动生成的审计日志，用户无需也不能手动创建或编辑工作记录。在工作项详情页底部默认为收起状态，避免干扰主体内容展示；仅在用户主动展开时才加载数据，减少不必要的接口请求。

## 数据关系

```
User (Account服务)  1 ──── N  工作项 (WorkItem)
                                  │
                                  ├──── 1 ──── N  子任务 (SubTask)
                                  │
                                  └──── 1 ──── N  工作记录 (WorkRecord)
```

- 一个用户可以有 0 到多个工作项
- 一个工作项可以有 0 到多个子任务
- 一个工作项可以有 0 到多个工作记录
- 子任务和工作记录的生命周期跟随所属工作项：删除工作项时级联删除其子任务和工作记录
- 所有数据通过 `user_id` 进行用户间隔离，用户只能访问自己的数据

## 界面概述

### 主页
- 工作项列表，以卡片或列表形式展示
- 支持搜索和排序
- 每个工作项卡片显示：标题、完成状态（已完成则显示删除线）、时间范围、子任务完成进度（如 3/5）
- 快速操作入口：新建工作项

### 工作项详情页
- 顶部：标题、完成状态切换、起止时间、提醒设置
- 中部：详细说明（支持富文本编辑）
- 子任务区域：子任务列表，每项前有勾选框
- 底部：工作记录（收起状态，点击展开后加载）

### 子任务区域
- 内嵌在工作项详情页中
- 每条子任务：勾选框 + 文本内容 + 起止时间 + 提醒图标
- 支持行内编辑
- 已完成子任务默认使用删除线样式

---

## 提醒机制

- 工作项和子任务均可独立设置提醒
- 提醒方式（一期）：浏览器通知 (Web Notification API)
- 提醒时机：到达设置的时间点时触发
- 后续可扩展：邮件提醒、微信推送等

## 用户系统

本系统不自行管理用户账户，而是接入统一的 Account 服务 (`https://account.takemeto.icu`) 作为身份认证中心。详情参见 [docs/用户系统.md](docs/用户系统.md)。

### 架构概览

```
Web App  ──OAuth2──┐
                   ├──> Account 服务 ──> 返回 access_token
Mobile App ─OAuth2─┘         │
                             │ 校验 token (/oauth/userinfo)
                             ▼
                    work_trace API ──> SQLite (按用户隔离数据)
```

### 接入方式

1. 在 Account 服务中注册 work_trace 为一个 OAuth 应用，获得 `client_id` 和 `client_secret`
2. 用户在 Web App 或 App 内通过 OAuth2 Authorization Code 流程登录，Account 服务返回 `access_token`
3. work_trace API 收到请求时，调用 Account 服务的 `/oauth/userinfo` 校验 token 并获取用户身份
4. work_trace 根据用户身份 (`sub`) 进行数据隔离，每个用户只能访问自己的数据

### 两种客户端认证

| 客户端 | 登录方式 | API 认证方式 |
|--------|----------|-------------|
| **网页 App** | 浏览器跳转 OAuth2 授权页面，用户登录并授权后重定向回 App | `Authorization: Bearer {access_token}` |
| **移动 App** | 同上，在 App 内嵌 WebView 完成 OAuth2 授权流程 | `Authorization: Bearer {access_token}` |

### 数据隔离

所有核心数据（工作项、子任务、工作记录）均关联 `user_id`（对应 Account 服务中的 `sub`），API 层强制过滤，确保用户只能操作自己的数据。

- 工作项、子任务、工作记录均增加 `user_id` 字段
- API 从 Bearer token 中解析 `sub` 作为当前用户标识
- 查询和写入时自动按 `user_id` 过滤，无需前端传递

### Token 管理

| 属性 | Access Token | Refresh Token |
|------|-------------|---------------|
| 格式 | JWT | 随机字符串 |
| 有效期 | 1 小时 | 30 天 |
| 用途 | 调用 work_trace API | 刷新 Access Token |
| 存储(Web) | 内存/localStorage | localStorage |
| 存储(App) | 安全存储(Keychain/Keystore) | 安全存储 |

客户端在 access_token 过期时，使用 refresh_token 通过 Account 服务刷新，无需用户重新登录。
